# WORKFLOW: product-definition

This workflow defines how the system should move from clarified context into stable product definition, design-ready structure, and bounded execution preparation.

It is the main macro workflow for converting a legitimate idea into bounded governed scope.

---

## Purpose

Use this workflow to turn clarified understanding into stable product artifacts that can later support safe execution.

This workflow exists to improve:

- scope clarity
- product boundary definition
- upstream external grounding
- experience and strategy maturity when materially required
- design-system maturity before downstream validation and execution preparation
- validation thinking
- disciplined task planning
- handoff readiness
- workflow-disciplined progression from new project entry to pre-implementation readiness

---

## Typical Governed Span

This workflow usually covers the core progression path through:

- `context`
- `research`
- `experience`
- `strategy`
- `prd`
- `design`
- `validation`
- `tasks`
- `handoff`

Not every project requires the same depth in every domain.

However, this workflow must still enforce explicit traversal of materially relevant domains rather than silently bypassing them.

For `New Project`, this workflow is the default governing workflow entry unless governance or state clearly requires another valid entry path.

---

## Workflow Traversal Rule

This workflow is not decorative documentation.

When it governs the current execution line, traversal through this workflow must be material and explicit.

That means:

- the controlling bounded skill must operate inside this workflow
- materially relevant upstream and downstream domains must be explicitly evaluated
- required specialist participation must be handled according to workflow need
- synthesis and structured outputs must be produced before advancing
- design-sensitive products must not silently skip the design layer once that layer is materially required

This workflow must not be treated as satisfied merely because one or more downstream skills happened to run.

---

## Canonical Skills

The main bounded skills associated with this workflow are:

- `discover-missing-context`
- `structured-research`
- `define-experience-direction`
- `define-growth-strategy`
- `define-prd`
- `define-design-foundation`
- `define-success-scenarios`
- `plan-tasks`
- `prepare-handoff`

These remain distinct bounded operations.

This workflow does not collapse them into one step.

These skills execute inside this workflow.

They do not replace it.

---

## Material Domain Evaluation Rule

The controlling skill inside this workflow must explicitly evaluate whether the following domains are materially needed before safe progression:

- `research`
- `experience`
- `strategy`
- `design`

A silent omission is not enough when one of these domains is materially relevant.

A correct bounded step inside this workflow should be able to state, honestly, one of the following for each materially relevant domain:

- this domain is required now and has been entered
- this domain is not materially required now
- this domain may be required later, but not yet

For `New Project`, `research` must be treated as required workflow work before PRD creation.

For user-facing products, `experience` and `strategy` must be explicitly evaluated before PRD safety is declared.

For projects whose downstream quality materially depends on shared user-facing surfaces, interaction-state coherence, accessibility materialization, or implementation-facing design contracts, `design` must also be explicitly evaluated before validation, task planning, handoff preparation, or implementation-readiness claims rely on that coherence.

This workflow must not silently interpret “not opened yet” as “not needed.”

---

## Research Rule for New Project

When the current path began through `Start a new project` / `New Project`, this workflow must not proceed directly from raw contextual discovery to PRD creation.

For `New Project`, structured research must be materially traversed before PRD.

This means:

- `research` is required, not optional, in the early portion of this workflow
- the relevant research specialists must be invoked through canonical agent resolution
- research findings must be synthesized explicitly
- a durable `Research` artifact must be produced in `project/research/active/` before PRD creation
- PRD readiness must not be declared until this research pass has materially completed

This rule exists to produce richer, more grounded product definition rather than shallow PRD generation from minimally clarified context.

Recognition of research need is not completion of research traversal.

Research completion is not permission to create the PRD in the same turn.

---

## Experience and Strategy Upstream Rule

For user-facing products, this workflow must explicitly evaluate whether:

- `experience-direction` is materially required before PRD
- `growth-strategy` is materially required before PRD

This evaluation must not be decorative.

It should consider whether safe progression materially depends on:

### Experience-direction signals

- interface direction
- information architecture
- trust-sensitive interaction
- visual hierarchy
- direction of system feel and quality bar
- motion or interaction tone
- page or surface structure

### Growth-strategy signals

- positioning
- messaging clarity
- CTA structure
- discoverability
- funnel logic
- analytics needs
- SEO or CRO significance
- strategic differentiation that materially shapes product framing

A correct progression path may still decide that one or both domains can open later.

But that conclusion must be explicit and honest.

This workflow must not bypass these domains silently when they are materially relevant to safe product framing.

---

## PRD Stabilization Rule

`define-prd` remains the canonical bounded step for product definition.

PRD remains the canonical product-definition artifact.

This workflow must preserve that role.

Accordingly:

- `experience-direction` does not replace PRD
- `growth-strategy` does not replace PRD
- `research` does not replace PRD
- `design-foundation` does not replace PRD

These domains may materially inform PRD safety and quality.

They do not replace the bounded product-definition step.

A PRD should only be treated as ready when upstream materially required contextual, research, experience, and strategy work has been handled honestly for the current execution line.

---

## Design Layer Rule

When the current execution line materially depends on shared surface consistency, reusable system logic, interaction-state coherence, accessibility materialization, or implementation-facing design contracts, this workflow must explicitly open the `design` layer after PRD and before downstream validation or execution preparation claims rely on that coherence.

The canonical bounded step for this layer is:

- `define-design-foundation`

This workflow must treat `design-foundation` as:

- a project-level design system artifact
- a governed layer after PRD
- the place where design becomes implementation-capable without collapsing into code

This workflow must not:

- treat `experience-direction` as though it already were the full design foundation
- skip the design layer silently when the project materially depends on it
- treat validation, task planning, or handoff preparation as fully mature when they still depend on unresolved design-foundation questions

For clearly user-facing digital products, `design` should normally be treated as materially relevant unless a governed equivalent already exists for the same execution line.

---

## Relationship Between Experience and Design

This workflow must preserve the architectural distinction between:

- `experience-direction`
- `design-foundation`

The distinction is:

- `experience-direction` defines upstream surface direction, tone, hierarchy, feel, and interaction intent
- `design-foundation` instantiates the project-level design system that makes those decisions durable and implementation-capable

A correct workflow traversal may therefore state, honestly:

- experience was materially required and traversed
- experience was sufficient for direction but not sufficient for design-system instantiation
- design-foundation was therefore still required after PRD
- or design-foundation was explicitly unnecessary because a governed equivalent already existed

This workflow must not collapse those two layers into one vague “design happened” claim.

---

## Specialist Participation Rule

Relevant specialists may include:

- `research`
- `architect`
- `security` when trust boundaries matter early
- relevant design specialists when user-facing experience direction materially affects product framing
- relevant growth specialists when messaging, positioning, discoverability, or funnel logic materially affects safe scope framing
- relevant design-foundation specialists after PRD when design-system maturity materially affects downstream quality

These specialists are not workflow controllers.

They remain bounded support inside the governing workflow.

However, when their participation is materially required for safe product-definition maturity, they must not be silently bypassed.

For `New Project`, the relevant research specialists must be invoked as part of the structured research pass before PRD.

For user-facing products, relevant design and growth specialists must also be explicitly evaluated rather than ignored by default.

For design-sensitive products, the specialists needed by `define-design-foundation` must be evaluated explicitly rather than being treated as “already covered” by direction alone.

---

## Required Synthesis Behavior

This workflow must produce synthesis, not just fragmented intermediate outputs.

At minimum, the workflow should materially produce:

- contextual synthesis
- research synthesis when research is required
- experience-direction synthesis when experience is required
- strategy synthesis when strategy is required
- product-definition synthesis through PRD
- design-foundation synthesis when design is required
- validation synthesis through bounded success scenarios
- execution-preparation synthesis through bounded tasking and handoff

A stage or domain should not be treated as mature merely because raw notes, partial artifacts, or specialist fragments exist.

Structured synthesis is part of valid traversal.

---

## Stage Completion Pause Rule

This workflow must preserve explicit pause between bounded macro moments.

Successful completion of one governed step inside this workflow must not automatically trigger the next one in the same turn.

This applies especially across boundaries such as:

- `context → research`
- `research → experience`
- `experience → strategy`
- `strategy → prd`
- `prd → design`
- `design → validation`
- `tasks → handoff`

The result of each bounded step should be:

- the artifact or synthesis for that step materially complete when required
- `PROJECT_STATE.md` coherently updated when valid
- the next allowed governed move made explicit
- return of control

This workflow must not compress its span into one continuous autonomous run.

---

## Expected Output

Across correct bounded traversal of this workflow, the system should eventually have:

- sufficiently clarified context
- a durable `Research` artifact when required
- a durable `Experience` artifact when required
- a durable `Strategy` artifact when required
- an active PRD or equivalent stabilized product definition
- a durable `Design Foundation` artifact when required
- bounded validation scenarios
- a bounded task plan
- a bounded handoff ready for possible execution unlock
- explicit workflow-visible acknowledgment of which materially relevant domains were traversed, deferred, or found unnecessary

These outputs belong to the workflow span as a whole.

They must not be interpreted as outputs that may be generated in one turn or one bounded governed step.

Each bounded step inside this workflow must still close independently, update state coherently when valid, return control, and stop before any later governed step begins.

For user-facing products, successful traversal should also leave visible whether `experience`, `strategy`, and `design` were:

- traversed now
- explicitly deferred
- explicitly found unnecessary for the current bounded moment
- already governed by an existing durable equivalent

---

## Readiness Signal

This workflow is sufficiently mature for execution preparation only when:

- the problem and boundary are explicit enough
- required research has been materially completed when applicable
- materially relevant experience and strategy needs have been explicitly handled honestly
- the PRD is bounded and stable enough
- the design layer has been materially handled when downstream coherence depends on it
- the validation artifact expresses real success scenarios
- tasks are decomposed without pretending parallel execution
- the active handoff defines the next bounded execution step honestly

This workflow must not claim maturity merely because downstream artifacts exist.

Maturity requires honest workflow traversal, synthesis, and bounded outputs.

When downstream execution preparation would still depend on an unresolved technical baseline, the system may require a bounded technical-baseline decision step before honest handoff preparation or implementation readiness can be claimed.

This does not create a new macro stage in the product-definition workflow.

It is a bounded governed decision step used only when baseline authority remains materially open.

---

## Must Not

This workflow must not:

- treat vague ideas as ready PRDs
- skip required structured research for `New Project`
- treat workflow traversal as satisfied by isolated skill execution alone
- silently bypass materially relevant domain work
- treat experience-direction as a substitute for PRD
- treat experience-direction as a substitute for design-foundation
- skip the design layer when downstream quality materially depends on it
- skip validation discipline
- turn task planning into implementation planning theater
- invent missing technical baselines during handoff preparation
- silently widen scope during decomposition
- treat research completion as permission to open PRD inline in the same turn
- treat user-facing experience, strategy, or design needs as implicitly resolved just because tasking is already possible
