# define-design-foundation

## Purpose

Use this skill to create a project-level design foundation artifact after product definition is stable enough to support a concrete, implementation-aware design system contract.

This skill exists to transform upstream direction into a project-specific design foundation that can guide implementation without repeated reinterpretation.

It sits between product definition and downstream validation / planning.

It must not behave like a branding brainstorm, a UI moodboard, or a screen-by-screen implementation handoff.

---

## What this skill is for

Use this skill when the system already knows enough about the product to define:

- a concrete token system for the project
- component families and state rules
- surface contracts for the main product surfaces
- structural accessibility constraints
- an implementation contract that frontend can consume with low drift

This skill should produce a strong project artifact.

It should not produce generic design advice.

---

## When to use

Use this skill only when all of the following are materially true:

- a stable PRD already exists
- a valid experience-direction artifact already exists
- a growth-strategy artifact exists if messaging, discovery, conversion, hierarchy, or action design materially affect the foundation
- the project has enough scope clarity to identify major surfaces
- the system is ready to stabilize design behavior before validation / task planning / implementation

In most product interfaces, this skill belongs after PRD and before validation and task planning.

---

## When not to use

Do not use this skill:

- during discovery
- before PRD is stable
- when experience direction is still vague or missing
- when the project still lacks clear surfaces or interaction expectations
- to generate final UI screens
- to replace implementation handoff
- to silently decide technical baseline or stack details that were not already authorized upstream

If inputs are not stable enough, stop and route back to the correct earlier step.

---

## Required inputs

This skill expects, at minimum:

- one active PRD artifact
- one active experience-direction artifact
- one active growth-strategy artifact when materially relevant
- the global design rules capability, once formally available
- project context sufficient to identify major surfaces, interaction patterns, and constraints

Primary input paths are expected to be:

- `project/prd/active/...`
- `project/experience/active/...`
- `project/strategy/active/...` when material
- `system/design/design-rules.md` once that capability exists canonically

If the design-rules capability is not yet formalized, use the best currently approved design architecture guidance available and explicitly state that the global rules layer is still pending formalization.

---

## Required specialists

### Mandatory specialists

- `design-system`
- `ui`
- `information-architecture`
- `frontend`

### Conditional specialists

- `motion`
- `branding`
- `page-strategy`

Frontend is not only a validator here.

Frontend is a co-author of the implementation contract portion of the foundation.

Its role is especially important for:

- naming viability
- output format viability
- theme strategy viability
- component mapping viability
- downstream drift reduction

---

## Specialist execution rules

When specialist execution is materially expected, this skill must:

- resolve specialists through the registry
- respect runtime type and invocation mode
- use the correct runtime call target
- distinguish materially succeeded vs partial vs failed vs unavailable
- never narrate routing failure or fallback synthesis as strong specialist execution

If specialist execution is unavailable, say so explicitly and downgrade confidence appropriately.

Do not fake specialist-backed authority.

---

## Canonical output

This skill must produce a project-level design foundation artifact.

Canonical target path:

- `project/design/active/DESIGN-FOUNDATION-001.md`

The artifact family is expected to live under:

- `project/design/active/`
- `project/design/done/`
- `project/design/archived/`
- `project/design/templates/`

The artifact must conform to the canonical structure defined by:

- `project/design/templates/DESIGN_FOUNDATION_TEMPLATE.md`

If the template is missing, this skill must stop and surface the missing prerequisite instead of improvising a non-canonical artifact shape.

---

## Minimum valid output rule

A design foundation generated by this skill is only valid if it materially contains:

- project-specific purpose
- design intent grounded in real product context
- token system
- component families
- variant and state model
- surface contracts
- accessibility materialization
- mandatory implementation contract
- versioning and change policy
- anti-entropy rules

If these elements are missing, do not present the artifact as complete.

It is only an incomplete direction artifact.

---

## Skill workflow

### 1. Validate entry conditions

Confirm that the PRD exists and is stable enough.

Confirm that experience direction exists and is materially usable.

Confirm whether growth strategy is materially relevant for this foundation.

Confirm that the project has enough surface clarity to define real contracts.

If these conditions are not met, stop and report the missing prerequisite.

### 2. Load upstream artifacts

Read the active PRD.

Read the active experience-direction artifact.

Read the active growth-strategy artifact when materially relevant.

Read the global design-rules layer if it already exists.

Use only the subset of inputs that materially shape the project foundation.

Do not pull unrelated design opinions into the artifact.

### 3. Identify material surfaces and system boundaries

Determine which product surfaces require explicit contracts.

Examples may include:

- landing
- dashboard
- editor
- public page
- checkout
- settings
- transactional flows

Only include surfaces that actually matter to the project.

Do not generate a fake universal product map.

### 4. Run specialist-supported synthesis

Use the required specialists to shape:

- token logic
- typography and hierarchy
- component families
- state model
- surface contracts
- accessibility materialization
- motion rules when relevant
- implementation contract viability

The synthesis must stay project-specific.

It must not turn into generic design-system doctrine.

### 5. Co-author the implementation contract

Treat the implementation contract as a mandatory foundation section, not an optional appendix.

This part must be co-authored with frontend-oriented reasoning.

It must define, at minimum:

- output formats
- naming convention
- theme strategy
- component mapping strategy
- implementation-facing state model
- implementation-facing surface contracts
- accessibility enforcement expectations
- versioning and change policy
- anti-entropy rules

This section must answer:

How does the design system move from artifact to code without repeated reinterpretation?

### 6. Write the artifact

Create the design foundation artifact in the canonical project design path.

Use the canonical template structure.

Do not collapse major sections just to make the output shorter.

Do not omit the implementation contract.

### 7. Report quality and uncertainty honestly

Report:

- what was materially defined
- which specialists materially contributed
- what remains unresolved
- whether the artifact is complete or only partially complete
- what assumptions were made

Do not claim operational completeness if major sections are still weak.

---

## State handling rule

This skill must remain honest about current governance maturity.

If the system has not yet formally incorporated the design phase or design artifact family into canonical governance/state rules, do not fake state alignment.

In that case:

- produce the artifact
- report the governance gap explicitly
- do not pretend the runtime and state model are already fully updated

Only update project state as if `design` were canonical when the state schema and governing rules have actually been updated to support it.

This prevents semantic drift between filesystem reality and state reality.

---

## What this skill must not do

This skill must not:

- replace the PRD
- introduce new product scope
- behave like a final UI spec
- behave like a screen-by-screen delivery plan
- choose a stack or framework that was not already authorized upstream
- skip accessibility materialization
- skip the implementation contract
- treat tool or routing fallbacks as strong specialist execution
- claim the foundation is complete when it is only partially structured

---

## Completion criteria

This skill is complete only when:

- a canonical design foundation artifact exists in the correct project path
- the artifact satisfies the minimum validity rule
- implementation contract is materially present
- unresolved items are explicitly surfaced
- specialist execution is reported honestly
- any governance/state mismatch is surfaced instead of hidden

---

## Handoff expectation

Once this skill completes successfully, the next canonical downstream step is normally:

- `define-success-scenarios`

However, if the design foundation reveals major upstream instability, route back to the correct earlier step instead of pushing forward mechanically.

The goal is not momentum.

The goal is stable downstream execution.
